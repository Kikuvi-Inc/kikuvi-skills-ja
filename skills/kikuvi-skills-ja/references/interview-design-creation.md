# ヒアリング設計作成ワークフロー
カスタマーヒアリング・エンゲージメントサーベイ・ユーザーリサーチなど、汎用的なヒアリング設計を行う際のガイド。

## フェーズ 1: 情報収集
1a と 1b は自由にイテレーションを回してよい。対話→リサーチ→再度対話、など必要に応じて繰り返す。全必須項目と学習目標が揃ったと判断したらフェーズ 2 へ進む。

### 1a. 対話による情報収集
以下のような情報をユーザーに質問して収集する。それ以外の観点も積極的に質問していき、ユーザーがどのような情報を集めたいかを引き出す。
- 目的の核心（例: 「最終的にどんな意思決定や打ち手につなげたいですか？」）
- 対象者の範囲（例: 「職種・在籍状況・人数規模を教えてください」）
- フォーカスエリア（例: 「特に深掘りしたいテーマや避けたい話題は？」）

### 1b. Web リサーチ
1a で得た目的・対象・フォーカスを踏まえて、業界・職種・既存リサーチの背景知識を WebSearch / WebFetch で補強する。デフォルトで検索を実行する。

ただし以下のいずれかに当てはまる場合は、検索を始める前にユーザーへテーマを確認する:
| 条件 | 理由 |
|------|------|
| 検索テーマやキーワードが一意に決まらない | 誤った方向での調査を避けるため |
| 業界・対象者像が複数の解釈を持ちうる | ユーザーの意図を確認してから着手するため |
| 社内独自のリサーチや非公開資料を参照する必要がありそう | Web では補完できない情報のため |

## フェーズ 2: ヒアリング設計の作成
フェーズ 1 が完了した後のみ実行する。
1. `design.create` の input schema を MCP から取得する
2. 得た情報をもとに `who / what / how / why` と `sections` を生成する
3. 作成後、デザインの URL をユーザーへ表示する（形式: `https://app.kikuvi.com/u/o/{orgId}/designs/{designId}`）

ユーザーは URL で実物を確認し、修正があれば都度指示する。テキストの draft を見せて承認を待つステップは不要。

## フェーズ 3: 修正対応
ユーザーが URL を確認して修正を依頼した場合、以下のエンドポイントで更新する。
| 修正内容 | 使用ツール | 制約 |
|---|---|---|
| タイトル・3W1H（who/what/how/why） | `design_update_metadata` | ACTIVE 後はタイトルのみ変更可 |
| セクション・質問の差し替え | `design_update_sections` | IN_DESIGN ステータスのみ |

修正後は変更箇所をユーザーへ簡潔に報告する。

## Critical Rule: ヒアリング設計作成の原則
- 即時実行の指示があっても、必須項目が揃っていない場合はユーザーへ確認してから進む
- 必要な情報を勝手に推測して補完しない。ユーザーに質問して情報を収集する
- Web リサーチは基本実施。ただし検索テーマが不明な場合は検索前にユーザーへ確認する
- Web の知識をそのまま使わず、ユーザー文脈に適応する
- 回答しやすさと分析可能性を両立する
